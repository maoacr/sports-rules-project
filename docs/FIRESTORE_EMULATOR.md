# Firestore Local Emulator

This project uses the [Firebase Local Emulator Suite](https://firebase.google.com/docs/emulator-suite) for local development. The emulator runs Firestore, Auth, and the Emulator UI on `localhost` — no live Firebase project required, no network round-trips, no cost.

## One-time setup

```bash
npm install -g firebase-tools   # if you don't have it yet
```

That installs the `firebase` CLI. Everything else in this repo is already configured:

| File | Purpose |
|---|---|
| `firebase.json` | Emulator ports (Firestore 8080, Auth 9099, UI 4000) and rule/index paths |
| `.firebaserc` | Project alias `demo-sports-rules` |
| `firestore.rules` | Security rules: public catalog read, user data private |
| `firestore.indexes.json` | Composite indexes for `searchArticles` and `getArticles` |
| `scripts/start-emulator.sh` | Convenience wrapper to start the suite |
| `scripts/import_seed.mjs` | `--emulator` flag to point the import at `localhost:8080` |

## Daily workflow

**Terminal 1 — start the emulator:**

```bash
./scripts/start-emulator.sh
# or: firebase emulators:start --only firestore,auth,ui --project demo-sports-rules
```

You'll see logs from Firestore, Auth, and the Emulator UI. Wait for the line:

```
┌─────────────────────────────────────────────────────────┐
│ ✔  All emulators ready! It is now safe to connect your app. │
└─────────────────────────────────────────────────────────┘
```

Open <http://localhost:4000> in your browser to browse the data once you seed it.

**Terminal 2 — seed the data:**

```bash
node scripts/import_seed.mjs --emulator --dry-run   # validate only
node scripts/import_seed.mjs --emulator            # real import
```

You should see 98 documents written (1 sport + 17 laws + 80 articles). Refresh the Emulator UI to see them.

**Terminal 3 — run the app:**

```bash
flutter run --dart-define=USE_FIRESTORE_EMULATOR=true
```

The app's `main.dart` reads the flag at compile time and calls `FirebaseFirestore.instance.useFirestoreEmulator('localhost', 8080)`. No code change, no env var at runtime — the flag is baked in.

To confirm the flag is active, look for the line in the Flutter console:

```
[emulator] Firestore pointing at localhost:8080
```

(If we add a `debugPrint` for it — not strictly necessary since the emulator UI will show the traffic.)

## Security rules

`firestore.rules` implements option **B — minimal realistic from day one**:

| Path | Read | Write |
|---|---|---|
| `sports/...` (catalog) | `if true` (public) | `if false` (admin SDK only, via the import script) |
| `users/{uid}/...` (user data) | `if request.auth.uid == uid` | same |

The Firestore emulator honors these rules. The import script uses the Admin SDK which bypasses them, so seeding always succeeds. The Flutter app uses the regular client SDK and is subject to the rules.

## Connecting the app to a different backend

| Scenario | Command |
|---|---|
| Local emulator (default for dev) | `flutter run --dart-define=USE_FIRESTORE_EMULATOR=true` |
| Live Firebase project (production) | `flutter run --dart-define=FIRESTORE_PROJECT_ID=my-real-project` |

The current `main.dart` only checks the emulator flag. To switch to a real project, the env var simply isn't set and the app uses whatever `firebase_options.dart` (or `Firebase.initializeApp()` defaults) points at. The Firebase options file is generated by `flutterfire configure` and is **not committed** — the emulator works without it because `Firebase.initializeApp()` falls back to default project resolution from the platform config, which in dev is `demo-sports-rules`.

## Cleaning state

The emulator stores its data in `.firebase/` (gitignored). To wipe and reseed:

```bash
rm -rf .firebase/
./scripts/start-emulator.sh   # restart
node scripts/import_seed.mjs --emulator
```

## Troubleshooting

**`Connection refused` on app start** — the emulator isn't running. Start it in another terminal.

**Permission denied when reading `sports/...` from the app** — the emulator is running but the rules file is wrong. Check `firebase.json` references `firestore.rules` and that the rules are in the file.

**Import script fails with "Could not load the default credentials"** — you ran without `--emulator` and without `GOOGLE_APPLICATION_CREDENTIALS`. Either provide a service account JSON (production) or pass `--emulator` (dev).

**The Emulator UI shows no data** — the import script didn't actually write. Re-run with `--dry-run` first to see what would be written, then without to commit.

## Why not use `firebase emulators:start` directly without these scripts?

You can. The `start-emulator.sh` script is just a thin wrapper that:
1. Sets the project alias (`demo-sports-rules`).
2. Restricts to `firestore,auth,ui` (no Cloud Functions, no Pub/Sub, no Storage — we don't need them yet).
3. Gives a one-line reminder to open the UI.

If you prefer, run the raw command instead — both work.
